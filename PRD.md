# PRD: `odin-my-wc`

## 1. Overview

`odin-my-wc`는 Odin으로 만드는 학습용 CLI 도구다. 목적은 Unix `wc`의 핵심 기능을 작은 범위로 재구현하면서, Odin의 기본 문법, 파일 I/O, 문자열 처리, 테스트 루틴, `defer` 사용 습관을 익히는 것이다.

이 프로젝트는 "작게 시작해서 끝까지 완성하는 것"을 우선으로 한다. v1에서는 단일 파일 입력만 지원하고, 출력 규칙과 오류 처리 방식을 단순하고 일관되게 유지한다.

## 2. Goal

- Odin으로 작은 CLI 프로그램 1개를 완성한다.
- `odin run`과 `odin test` 중심의 개발 루틴을 정착시킨다.
- 파일 읽기, 텍스트 스캔, 공백 기준 단어 계산, 줄 수 계산을 직접 구현한다.
- 최소 10개의 테스트 케이스로 동작을 검증한다.
- 리소스 정리와 에러 처리 흐름에서 `defer`를 의식적으로 사용한다.

## 3. Non-Goals

다음 항목은 v1 범위에서 제외한다.

- 디렉터리 재귀 탐색
- 여러 파일 동시 입력
- 표준 입력(stdin) 지원
- 색상 출력
- 성능 최적화
- 멀티스레드 처리
- GNU `wc`와의 완전한 옵션 호환
- 로케일/유니코드 단어 경계의 정교한 지원

## 4. Target User

- Odin을 처음 프로젝트 형태로 익히는 개발자
- CLI 프로그램 구조와 테스트 작성 루틴을 연습하려는 학습자

## 5. Problem Statement

Odin 입문 단계에서는 문법만 읽고 넘어가면 실제로 손에 남는 것이 적다. 반대로 너무 큰 프로젝트를 고르면 언어 학습보다 구조 설계와 디버깅에 시간을 다 쓰게 된다.

`odin-my-wc`는 아주 작은 문제를 명확한 입출력 규칙으로 풀면서, 다음 역량을 빠르게 체득하기 위한 프로젝트다.

- 파일을 안전하게 읽는 흐름
- 텍스트를 순회하며 상태를 계산하는 흐름
- 실패 케이스를 분리해 처리하는 흐름
- 테스트 가능한 순수 로직과 CLI 입출력 로직을 분리하는 흐름

## 6. Product Scope

v1은 텍스트 파일 1개를 입력받아 다음 값을 출력한다.

- 줄 수 (`lines`)
- 단어 수 (`words`)
- 바이트 수 (`bytes`)
- 입력 파일 경로 (`path`)

기본 출력 형식은 아래와 같다.

```text
<lines> <words> <bytes> <path>
```

예시:

```text
12 53 301 sample.txt
```

## 7. User Stories

- 사용자는 파일 경로를 넘겨서 해당 파일의 줄 수, 단어 수, 바이트 수를 확인할 수 있다.
- 사용자는 존재하지 않는 파일 경로를 넣었을 때, 프로그램이 조용히 실패하지 않고 명확한 오류 메시지를 볼 수 있다.
- 개발자는 카운트 계산 로직을 함수 단위로 테스트할 수 있다.
- 개발자는 CLI 파싱과 핵심 계산 로직이 분리된 구조를 통해 변경 영향을 쉽게 통제할 수 있다.

## 8. Functional Requirements

### 8.1 Input

- 프로그램은 파일 경로 1개를 입력으로 받는다.
- v1에서는 파일 경로가 없거나 2개 이상이면 사용법 또는 오류 메시지를 출력하고 종료한다.
- 초기 구현은 위치 인자 1개만 지원해도 된다.
- 확장 가능성을 고려해 이후 `core:flags` 기반 옵션 구조로 옮길 수 있게 설계한다.

### 8.2 Output

- 성공 시 한 줄로 결과를 출력한다.
- 기본 출력 순서는 `lines words bytes path`다.
- 숫자는 공백으로 구분한다.
- 경로는 입력받은 문자열 그대로 출력한다.

### 8.3 Counting Rules

#### Bytes

- 바이트 수는 파일을 읽은 원시 바이트 길이로 계산한다.
- 텍스트 인코딩 해석 결과가 아니라 파일 내용의 실제 바이트 수를 기준으로 한다.

#### Lines

- 줄 수는 개행 문자 `\n`의 개수로 계산한다.
- 마지막 줄에 개행이 없으면 추가로 1줄을 보정하지 않는다.
- 즉, v1은 "newline count" 규칙을 따른다.

#### Words

- 단어 수는 ASCII 공백류를 기준으로 계산한다.
- 공백 문자는 최소한 다음을 포함한다: space, tab, newline, carriage return.
- 연속된 공백은 하나의 구분자로 취급한다.
- 공백이 아닌 문자의 구간 하나를 단어 1개로 센다.
- 한글 등 비ASCII 문자가 포함되어도, 공백 기준 구분 원칙은 동일하게 적용한다.

### 8.4 Error Handling

- 파일을 열거나 읽을 수 없으면 오류 메시지를 출력하고 non-zero exit code로 종료한다.
- 인자가 잘못되면 오류 메시지를 출력하고 non-zero exit code로 종료한다.
- panic에 의존하지 않고 예측 가능한 오류 흐름을 사용한다.

## 9. CLI Contract

v1 CLI는 아래처럼 동작한다.

```text
odin-my-wc <path>
```

에러 예시:

```text
usage: odin-my-wc <path>
```

또는

```text
error: failed to read file: <path>
```

v1.1 확장 옵션은 아래를 목표로 하지만 이번 구현 범위에는 포함하지 않는다.

```text
odin-my-wc --lines --words --bytes <path>
```

## 10. UX Principles

- 출력은 짧고 예측 가능해야 한다.
- 사용자는 성공/실패를 한눈에 구분할 수 있어야 한다.
- 학습 프로젝트이므로 내부 로직이 읽기 쉬운 구조여야 한다.
- CLI 겉모습보다 테스트 가능성과 구조적 단순함을 우선한다.

## 11. Technical Design Constraints

- 구현 언어는 Odin이다.
- 표준/코어 패키지 사용을 우선한다.
- 핵심 계산 로직은 가능한 한 순수 함수에 가깝게 분리한다.
- CLI 입력 처리와 파일 읽기, 카운트 계산, 출력 포맷을 분리한다.
- 파일/리소스 정리 지점에는 `defer`를 적극 사용한다.
- 테스트에서 직접 검증 가능한 함수 시그니처를 우선한다.

## 12. Suggested Module Breakdown

권장 구조는 아래와 같다.

- `main.odin`: 프로그램 진입점, 인자 검증, 오류 처리, 결과 출력
- `wc.odin`: 카운트 계산 로직
- `wc_test.odin`: 순수 계산 로직 테스트
- `README.md`: 실행 예시와 범위 설명

권장 주요 타입:

```text
Counts {
  lines: int
  words: int
  bytes: int
}
```

권장 주요 함수:

- `count_bytes(data) -> int`
- `count_lines(data) -> int`
- `count_words(data) -> int`
- `count_all(data) -> Counts`
- `read_file(path) -> data | error`

## 13. Acceptance Criteria

- 파일 경로 1개를 받아 결과를 출력한다.
- 빈 파일에 대해 `0 0 0 <path>`를 출력한다.
- 공백만 있는 파일에서 단어 수가 0으로 계산된다.
- 여러 줄 파일에서 줄 수가 `\n` 기준으로 일관되게 계산된다.
- 한글이 포함된 파일에서도 바이트 수와 공백 기준 단어 수가 일관되게 계산된다.
- 존재하지 않는 파일 입력 시 non-zero exit code로 종료한다.
- 테스트가 최소 10개 존재한다.
- `odin test`가 통과한다.

## 17. Risks

- 줄 수 규칙을 중간에 바꾸면 테스트와 출력 기대값이 흔들릴 수 있다.
- 단어 수 정의를 너무 넓게 잡으면 학습 범위를 벗어난다.
- 처음부터 `wc` 전체 호환을 노리면 프로젝트가 커진다.

- 단어 수는 공백 기반 상태 머신으로 계산
- 출력 순서는 `lines words bytes path`
- v1에서는 옵션 최소화

- 코드가 한 번 동작하는 수준이 아니라 반복 실행 가능한 CLI 형태다.
- 핵심 로직이 테스트로 보호된다.
- 출력 규칙이 문서화되어 있다.
- 다음 확장(`core:flags`, 다중 옵션 출력)으로 자연스럽게 이어질 수 있다.
